\section{Methodology \& Justifications}
\label{sec: Methodology}
This project implements an eight-stage supervised classification pipeline to create directional risk signals for Bitcoin. Daily input data is sourced from three providers: OHLCV price data from Yahoo Finance, on-chain metrics from CoinMetrics, and features engineered using raw OHLCV data and the ta.Lib Python module. The data is processed through sequential stages of Triple Barrier labelling, train/test splitting, feature selection, hyperparameter tuning, and model training for four classifiers: Logistic Regression, SVM-RBF, Random Forest, and XGBoost. Their performance is assessed against both statistical and trading metrics. The pipeline produces three-class outputs (+1 - Oversold, 0 - Neutral, -1 - Overbought) for a 7-day horizon. Intermediate outputs are stored locally as pickle files between stages, allowing each notebook to run independently, with all experiments logged via MLflow for version control. The full architecture is illustrated in Figure \ref{fig:meth Project Pipeline}.

% =============== DATA SOURCES AND ACQUISITION ===============
\subsection{Data Sources \& Acquisition (0.75 pages)}
\label{subsec: Data Sources}
Three distinct data sources were combined to capture price, sentiment, and on-chain behaviour. Publicly available sources were preferred for accessibility, lack of sources where data access was not restricted meant multiple data sources were necessary.
% =============== YAHOO FINANCE ===============
\subsubsection{Yahoo Finance (via yfinance)}
\label{subsubsec: yfinance}
Daily BTC-USD Open, High, Low, Close, Volume (OHLCV) data from 24/09/2014–01/03/2026 were downloaded via the yfinance library as a Pandas DataFrame. yfinance is an unofficial wrapper around Yahoo Finance's internal data feed rather than a licensed primary feed. This does not affect project implementation - yfinance is widely used in academic ML-finance research and daily OHLCV data is consistent across providers at this frequency \cite{chenAnalysisBitcoinPrice2023}.
% =============== COIN METRICS ===============
\subsubsection{CoinMetrics}
\label{subsubsec: CoinMetrics}
CoinMetrics is an open-source project founded in 2017 with a public community API. Thirty-one fundamental daily blockchain activity metrics were retrieved \cite{coinMetrics} from the first data available (03-01-2009). Metrics like MVRV, mean hash rate capture supply/demand dynamics and network health that price-only data misses. All features listed in Table \ref{tab:features_full} in Appendix \ref{app: Results}.
Other on-chain data providers were evaluated (e.g. BGeometrics, CoinGecko) but rejected due to rate limits on free tiers making systematic retrieval impractical for a reproducible pipeline.
% =============== FGI DATA ===============
\subsubsection{Fear \& Greed Index (fear-and-greed-crypto)}
\label{subsubsec: FGI}
The Fear \& Greed Index (FGI) was downloaded and cached from an unofficial Python wrapper \cite{reismanRhettreFearandgreedcrypto2026}. FGI availability is limited to 2018 onwards, which imposes a data constraint. This presents a tradeoff: reduce the dataset range by $\sim$4 years. The composite index captures volatility, momentum, and volume signals that are already captured by the Ta-Lib indicators. It was excluded due to partial feature redundancy. 
% =============== FEATURE ENGINEERING ===============
\subsubsection{Feature Engineering (Ta-Lib)}
\label{subsubsec: Feature Engineering}
Ta-Lib (Technical Analysis Library) is a C-based open-source library with Python bindings that computes standardised technical indicators from OHLCV inputs \cite{TALibTechnicalAnalysis}. In the project notebook ‘02 Feature Engineering’ (Figure \ref{fig:meth Project Pipeline}) it is used alongside Pandas/NumPy to construct the full feature matrix from the raw price history. The notebook produces 58 features in total over five categories: momentum, trend, volatility, volume, and trend direction.\\
\noindent Ta-Lib implements indicators according to their canonical published specifications (e.g., Wilder's original RSI definition, not a simplified rolling-mean approximation). It is also the dominant library in the academic ML-finance literature. Using Ta-Lib, along with following standard literature defaults for parameters, ensures that the project maintains comparability with other works \cite{radmandHybridTLBOXGBoost2025, shangSentimentDrivenBitcoinPrice2025, tummonTradingCryptocurrencyDeep2020}. Calculating indicators with lookback periods produces leading NaN rows, introducing a constraint on available data. 
% =============== DATA ALIGNMENT AND MERGING ===============
\subsubsection{Data Alignment \& Merging}
\label{subsubsec: Data Alignment \& Merging}
\textbf{Merging:} Each source was reindexed to a full daily DatetimeIndex covering its own native date range. Fundamental metrics were sequentially merged into the OHLCV DataFrame using direct column assignment based on the price history index. As confirmed by inspection, the only missing price values were the final day's finance OHLCV entries.\\
\noindent \textbf{Data cleaning:} Ta-Lib NaN warmup - These rows are explicitly dropped for the primary model training dataset in notebook ‘04 Data Label \& Split’. Excluded deterministic fields (total issuance, daily block count) and redundant USD-denominated duplicates of native fields from CoinMetrics data.\\
\textbf{Storage:} Intermediate data was serialised to pickle files at each pipeline stage (01\_bitcoin\_data.pkl, 02\_engineered\_features.pkl, etc.). This pattern decouples pipeline stages, enabling independent re-execution of any notebook without re-running upstream data acquisition.
% =============== DATA LABELLING ===============
\subsection{Data Labelling}

To implement an end-to-end pipeline before diving deep into the research and finalising how to label data, scipy.signal.find\_peaks \cite{Find_peaksSciPyV1170} was used to generate provisional binary labels marking local price maxima. This approach served as a functional placeholder.
% =============== THREE CLASS & TRIPLE BARRIER JUSTIFICATION ===============
\subsubsection{Justification for three classes \& Triple Barrier Approach}
A binary label (up/down) forces a directional position every day. In practice, many days lack clear opportunities. The Neutral class absorbs ambiguous observations, so the model only trades when there is a sufficiently strong signal. Furthermore, the Neutral class offers control over the up/down barriers, providing the versatility to tailor trade signals aligned with particular strategies. The triple barrier method addresses a key issue - labels reflect the path of prices, not solely the endpoint \cite{lopezdepradoAdvancesFinancialMachine2018}. Otherwise, a label assigned from today's close to next week's close disregards what happened intra-period.
% =============== TRIPLE BARRIER IMPLEMENTATION ===============
\subsubsection{Triple Barrier Implementation}
For each entry day $i$, two horizontal barriers and one vertical barrier are defined:
\begin{gather}
Profit\_target =  P_i \times (1 + \lambda_p \times \sigma_i)\label{eq: tp profit} \\
Stop\_loss =  P_i \times (1 + \lambda_s \times \sigma_i)\label{eq: tp loss}
\end{gather}
where $P_i$ is the closing price, $\sigma_i$ is the 100-day EWMA volatility at day $i$, and $\lambda_p$, $\lambda_s$ are the profit and stop multipliers respectively. The vertical barrier is a fixed holding period of 7 days.\\
EWMA is more responsive to recent volatility than a simple rolling standard deviation. A spike yesterday immediately shifts the barrier width rather than being diluted across the entire 100-day window. This means the barriers produce adaptive position sizing, which is meaningful for a real trading context, where they are wider during high-volatility regimes. The 100-day window is an adoption of the standard Lopez strategy \cite{lopezdepradoAdvancesFinancialMachine2018}.
The labels were assigned using the following decision logic:\\
$\bullet$ If profit target is hit first within the holding period: $+1$\\
$\bullet$ If stop loss is hit first: $-1$\\
$\bullet$ If neither barrier is hit within 7 days: 0\\
$\bullet$ If both are hit: whichever was hit at the earlier index position wins.\\
For each daily entry, the triple\_barrier.py (Figure \ref{fig:meth Project Pipeline}) outputs the label, returns, exit\_price, and barrier\_hit\_times. The return is the realised return from entry to whichever barrier was hit -- this feeds directly into the trade metrics evaluation in notebook ‘08 Model Evaluation’ (Figure \ref{fig:meth Project Pipeline}).
% =============== TRIPLE BARRIER PARAMETERS ===============
\subsubsection{Optimising Triple Barrier Parmameters}
The multipliers $\lambda_p$ and $\lambda_s$ determine the barrier width and therefore the entire label distribution. All downstream model performance is sensitive to this decision; different values produce:\\
$\bullet$ Different class proportions, class imbalance degrades performance of some classifiers \cite{lopezdepradoAdvancesFinancialMachine2018}.\\
$\bullet$ Different expected returns, which affect the performance of trading metrics.\\
\textbf{Parameter search approach:} the expected value per bar of +1 labels was used as the fitness value to optimise the parameters of the Triple Barrier Labeling Method \cite{fuEnhancedGeneticAlgorithmDrivenTriple2024}.
\begin{gather}
Objective =  f_{+1} \times \bar{r}_{+1} \label{eq: tp objective}
\end{gather}
It is a model-agnostic metric that rewards both finding profitable long opportunities frequently (frequency, $f_{+1}$) and making them meaningfully profitable (mean realised return, $\bar{r}_{+1}$) when they occur. To mitigate severe class imbalance, a normalised Shannon entropy lower threshold of 0.85 was set \cite{deduEntropyBasedPortfolioOptimization2026}. The threshold was determined empirically to achieve a sufficient proportion +1 labels while preserving class distribution. Any trial with entropy below this threshold was scored 0 and discarded.\\
Optuna is an open-source hyperparameter optimisation framework \cite{OptunaHyperparameterOptimization} used in timeseries ML literature to implement automatic bayesian search for specified search space \cite{stoikovMarketMakingCrypto2025, bieganowskiSupervisedAutoencoderMLP2025}. After the last $20\%$ of the dataset is dropped to remove the test set, a search space of $[0.5, 3.0]$ is used to find $\lambda_p/_s$ in steps of $0.05$. A Tree-structured Parzen Estimator (TPE) search algorithm with a random seed was configured for 100 trials to search for the multipliers that would maximise the objective \cite{ozakiMultiobjectiveTreeStructuredParzen2022}. All metrics were logged to MLflow, a widely adopted open-source Artificial Intelligence operations platform \cite{MLflowOpenSource}. The final multipliers and label distributions are presented in Table \ref{tab:res tp parameters}
% =============== TRAIN/TEST SPLIT===============
\subsection{Train/Test Split}
Splitting data with 80/20 ratio is a well-established convention for time-series financial ML with small datasets \cite{kurniasariHybridARIMAANNModel2025, oladeleDeepNeuralNetwork2023}. The 80/20 split gives $\approx 3,200$ training rows (04/04/2015-16/11/2022 and 24/11/2022-22/02/2026) – which is already limited. Dropping to 70/30 would yield $\approx 2,800$ training rows and a larger test set than necessary, which goes against the findings of Ngai et al. (2023), who found that prediction accuracy improved as the split increased from 70/30 to 90/10 \cite{eddiengaiTimeSeriesPrediction2023}.\\
The seven rows immediately after the training cutoff are labeled "Gap" and are excluded from both the train and test sets to protect against look-ahead contamination. While this implementation will have a negligible impact on performance for a 7-day horizon, it is crucial for potential future implementations comparing a 7-day with longer horizons.\\
The chronological 80/20 holdout is the primary evaluation scheme for simplicity of implementation when involved with serialised data between notebooks; walk-forward cross-validation (CV) is used only during hyperparameter search (Figure \ref{fig:meth Project Pipeline}) to prevent overfitting to the single held-out fold.
% =============== FEATURE Scaling ===============
\subsection{Feature Scaling}
Feature scaling was applied before model training to address the scale sensitivity of Logistic Regression and SVM. A mixed scaling strategy was adopted based on the distributional properties of each feature: bounded oscillators (RSI, Stochastic \%K, \%D) were using their theoretical limits. This prevents degraded model performance from clipped or mis-scaled feature values outside the training range, along with miscalibration associated with regime-specific training windows. Also, the zero in the scaled features is aligned with true zero for interpretability.\\
Volume-derived and price-derived features (MACD, EWMA volatility, EMA) were scaled using RobustScaler, which uses the median and interquartile range rather than the mean and standard deviation, providing resistance to the extreme outliers characteristic of Bitcoin during crash and rally periods \cite{RobustScaler}. While bounded features are scaled in notebook ‘04 Data Label \& Split’ as part of the dataset, unbounded features are scaled within the cross-validation pipeline during hyperparameter tuning to prevent leakage across CV folds.Then, a Robust Scaler was fitted exclusively to the training partition to prevent data leakage from the test set and applied before model training.\\
The code applies scaling uniformly to all models including the scale-invariant RF and XGBoost. A consistent preprocessing pipeline reduces the number of pipeline variants to test and avoids subtle bugs from applying different transformations to different model paths.
% =============== FEATURE SELECTION ===============
\subsection{Feature Selection}
Feature selection was applied to reduce model variance from irrelevant features and to ensure methodological rigor rather than to address computational cost, which was not a primary constraint at this dataset scale. SVM is most sensitive to the curse of dimensionality \cite{omoleUsingMachineDeep2025} because it scales poorly with uninformative dimensions, but Random Forest is largely self-regulating via max\_features \cite{RandomForestClassifier}.\textcolor{red}{In the absence of a priori domain knowledge, a data-driven approach was adopted to avoid subjective feature ranking: the dataset was allowed to determine relevance through a statistical selection procedure.}\\
Scoring features by their intrinsic statistical properties using a filter method was deemed insufficient for a model-centric evaluation where joint predictive power and feature interactions need consideration \cite{omoleUsingMachineDeep2025}. In wrapper methods, Boruta was chosen over RFE because it provides an intrinsic approach to feature selection without requiring a pre-specified number of features. Boruta establishes a noise baseline in each trial by creating shadow features – column-wise permutations of real features that eliminate predictive signal. Feature relevance is confirmed via a binomial test with Bonferroni correction, only if its importance consistently exceeds the maximum shadow importance across iterations \cite{kursaFeatureSelectionBoruta2010}.\\
Boruta was fitted exclusively on the training split to prevent test-set contamination. Parameters were set as follows: max\_iter=100 (default value), alpha=0.05 (default value), and perc=100, the most conservative setting, each feature must consistently outperform the single strongest shadow feature, not an average noise level. The internal Random Forest used a shallow tree max\_depth of 5 to prevent overfitting during importance scoring and class\_weight='balanced' to correct class imbalance. Tentative features were deemed statistically insignificant after inspecting Figures \ref{fig:res boruta importance balanced} and \ref{fig:res boruta importance max EV}. Feature selection was applied to both triple barrier configurations: equal label split and EV-optimised.
% =============== ML MODELS ===============
\subsection{ML Models}
Four classifiers were selected based on the following constraints of the pipeline: a dataset of approximately 4,000 daily observations, a three-class output (profit-hit, Neutral, loss-hit), and a flat feature matrix that excludes sequence-dependent architectures like LSTM or GRU. Having models that let us set class\_weight='balanced' to correct for class imbalance was desirable but not necessary.\\
Logistic Regression serves as the interpretable linear baseline, providing the performance floor against which non-linear models are evaluated \cite{ghoddusiMachineLearningEnergy2019}. With class\_weight='balanced' and the saga solver (which handles multiclass via softmax natively and scales to the dataset size), it provides a well-calibrated probabilistic output and is entirely transparent: every prediction is a linear combination of confirmed features \cite{LogisticRegression}.\\
SVM with an RBF kernel maps the input features into a high-dimensional space where non-linear decision boundaries become linear separable hyperplanes. This inherently regularised approach, which finds the maximum-margin hyperplane between classes, resists the curse of dimensionality, improving generalisation and overfitting performance, especially in high-dimensional settings \cite{renForecastingStockMarket2019, yuEvolvingLeastSquares2009}.\\
Random Forest is an ensemble of decorrelated decision trees, trained using bootstrap aggregation (bagging) and random feature subsets at each split (max\_features). This dual randomness in observations and features makes the model robust to overfitting on small datasets and genuinely resistant to irrelevant features, as no single tree can rely on spurious predictors consistently \cite{breimanRandomForests2001}.\\
XGBoost implements gradient-boosted trees with built-in L1 and L2 regularisation (reg\_alpha, reg\_lambda), making it well-suited to the multicollinear feature set (multiple indicators are derived from the same underlying price series). It is among the most consistently cited high-performing classifiers in financial forecasting benchmarks \cite{raoComparativeStudyStatistical2025, qureshiEvaluatingMachineLearning2025, eddiengaiTimeSeriesPrediction2023}.\\
Sequence models (LSTM, GRU) and deep architectures were excluded because the pipeline's flat daily feature matrix is incompatible with sequential input for sequence models, and data-hungry architectures are penalised at this dataset scale.
% =============== HYPERPARAMETER SELECTION ===============
\subsection{Hyperparameter Tuning}
Hyperparameters shape the model behavior and learning capability. This makes hyperparameter tuning a critical process for optimising machine learning model performance. The process involves searching the defined parameter space to find the combination that gives the best performance for a specific task. \cite{qureshiEvaluatingMachineLearning2025}.\\
Tree-structured Parzen Estimator (TPE) was selected over grid and random search because it uses past evaluations to build a probabilistic model of the objective surface, guiding subsequent samples toward promising regions. This efficiency is critical, as an exhaustive search across the multi-dimensional spaces of the four models would be computationally intensive, delaying progress given the limited available resources \cite{akibaOptunaNextgenerationHyperparameter2019}.\\
TimeSeriesSplit with a gap = 7-day-horizon was used to ensure that the full label-forming horizon is excluded from the boundary between training and validation, preventing the last training label's forward-looking window from overlapping with the first validation observation \cite{lopezdepradoAdvancesFinancialMachine2018}.\\
Macro-averaged F1 was chosen as the primary optimisation target because it assigns equal weight to each of the three signal classes regardless of their frequency, preventing Optuna from converging on models that simply over-predict the dominant class.
\subsubsection{Search Space Justification}
The regularisation parameter $C$ for Logistic Regression and SVM was searched over a logarithmic scale across eight orders of magnitude ($10^{-4}$ to $10^{4}$) \cite{RegularizationPathL1, hsuPracticalGuideSupport2003}.
Parameter value ranges for Random Forest and XGBoost were set empirically. No domain-specific study was identified that prescribes specific bounds for financial time series classification; the wide range ensures the search space is not artificially constrained. Gamma for SVC was defined on logarithmic scales, following the SVM parameter ranges outlined by Hsu et al. (2003) \cite{ScikitlearnExamplesSvm}.
% =============== FINAL MODEL TRAINING ===============
\subsection{Final Model Training}
Following hyperparameter optimisation, the best trial's parameters were loaded from MLflow and each classifier was retrained on the full training set. Cross-validation serves only to evaluate hyperparameter configurations during search -- it is not used for the final train/test split.

% =============== MODEL EVALUATION ===============
\subsection{Model Evaluation}
Classification metrics were computed on the held-out test set using the calculate\_ml\_metrics (Figure module. Matthews Correlation Coefficient (MCC) and balanced accuracy were logged as supplementary metrics to provide robustness against the known limitations of F1 under class imbalance \cite{chiccoAdvantagesMatthewsCorrelation2020}. 
The Calculate\_trade\_metrics module implements a long-only strategy: a trade is entered only when $y\_{pred} = +1$ (profit-target-hit), with the holding period return provided by the Triple Barrier label window of seven days. No position is taken on neutral or short signals, making the strategy unambiguous in its direction. The Sharpe ratio and Sortino ratio were both computed on annualised daily returns (periods=365). The Sortino ratio is preferred for Bitcoin's asymmetric return distribution because it only penalises downside volatility, unlike the Sharpe ratio, which implicitly penalises large positive returns \cite{DownsideRiskProQuest}. The avg\_loss\_when\_wrong metric captures the average return on trades where $y_{pred} = +1$ but the price did not rise—isolating the cost of false positive signals specifically. All model strategies are benchmarked against a passive Bitcoin buy-and-hold return computed over the same test period using QuantStats, providing a market-adjusted performance reference.
